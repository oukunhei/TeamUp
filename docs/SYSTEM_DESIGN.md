## 1. 产品概述

### 1.1 背景与目标

大学生在课程项目、竞赛、科研中常面临组队困难：
- 有想法但找不到志同道合的队友
- 有技能但找不到合适的项目
- 老师布置的小组作业需要快速组队
- 担心创意被抄袭，需要可信的时间戳证明

**TeamUp** 旨在构建一个"创意驱动的组队平台"，让"有想法的人"与"有技能的人"高效匹配。

### 1.2 核心定位

| 维度 | 描述 |
|------|------|
| **短期** | 大学生课程项目/竞赛组队工具 |
| **中期** | 校园创新项目孵化平台 |
| **长期** | 开源式项目协作平台（Walk-in 加入项目） |

### 1.3 产品哲学

- **开放式创新**: 创意公开，细节受控
- **可信存证**: 时间戳保护原创性
- **轻量协作**: 降低组队门槛，提高匹配效率
- **教育导向**: 适配课程场景，支持教学管理

## 2. 功能模块设计

### 2.1 空间体系（双轨制）

```
┌─────────────────────────────────────────────────────┐
│                    TeamUp 平台                       │
├─────────────────────────┬───────────────────────────┤
│      开放空间 (Open)     │      课程空间 (Course)     │
│  ─────────────────────  │  ───────────────────────   │
│  • 任何人可发布创意       │  • 老师/TA创建空间         │
│  • 跨专业自由组队         │  • 绑定具体课程与作业       │
│  • 无时间/课程限制        │  • 学生通过邀请码/学号加入   │
│  • 长期项目孵化           │  • 有明确的截止时间         │
│  • 类似"开源社区"         │  • 有评分/展示需求          │
└─────────────────────────┴────────────────────────────┘
```

#### 开放空间 (Open Space)
- **创意广场 (Idea Hub)**: 浏览所有公开的创意卡片
- **发起项目**: 填写创意概述、所需技能、预期成果
- **加入申请**: 向项目 Holder 发送加入请求
- **项目看板**: 组队成功后的协作空间

#### 课程空间 (Course Space)
- **空间管理**: 老师/TA 创建空间，设置课程信息、作业要求
- **成员管理**: 学生通过学号/邀请码加入
- **作业发布**: 老师发布具体的 Project 要求
- **强制组队**: 可设置组队人数限制、截止时间
- **成果提交**: 最终项目展示与评分（扩展）

### 2.2 创意 (Idea) 生命周期

```
[创建 Idea]
    │
    ▼
┌──────────────┐
│   草稿状态    │  → 仅创建者可见，可编辑
└──────────────┘
    │
    ▼ (发布)
┌──────────────┐
│   公开概述    │  → 所有人可见：标题、简介、标签、所需技能
└──────────────┘
    │
    ▼ (收到有意者申请)
┌──────────────┐
│   申请审批    │  → Holder 查看申请者资料，决定是否给予"只读权限"
└──────────────┘
    │
    ▼ (审批通过 → 有意者获得只读权限)
┌──────────────┐
│   项目内容    │  → 审批通过者可见：详细文档、项目进度、版本历史
└──────────────┘    (此页面防复制/截图，支持双向筛选)
    │
    ▼ (Holder 确认对方正式加入团队)
┌──────────────┐
│   执行阶段    │  → 核心成员：可查看详细文档、上传新版本、参与看板
└──────────────┘    (项目看板，自动化甘特图进度跟踪)
    │
    ▼ (组队完成，关闭申请功能)
    │
    ▼ (项目完成)
┌──────────────┐
│ 提交/展示成果 │  → 课程空间内可以提交，之后可选不公开/课程可见/公共空间可见
└──────────────┘  → 公共空间可选公开/不公开
```

**双向筛选机制**：
1. **申请者筛选项目**：通过公开概述判断项目是否匹配自己的技能和兴趣
2. **Holder 筛选申请者**：查看申请者的技能标签、个人简介，决定是否给予"只读权限"
3. **双方确认**：申请者查看详细文档和进度后，确认是否正式加入；Holder 最终确认其成为核心成员

### 2.3 权限模型

| 角色 | 开放空间 | 课程空间 |
|------|---------|---------|
| **访客** | 浏览公开概述 | 仅可看到空间存在（需登录） |
| **普通用户** | 发布Idea、申请加入、被邀请 | 加入空间、参与组队 |
| **审批通过者 (Viewer)** | 只读查看项目内容与进度 | 同左（课程空间内） |
| **项目Holder** | 管理自己的Idea、审批申请、编辑文档 | 同左 + 课程内权限 |
| **课程Owner** | - | 创建/管理空间、发布作业、查看所有组队 |
| **管理员** | 全平台管理 | 全平台管理 |

**权限层级（针对单个 Idea）**：
1. **公开概述** → 所有人
2. **只读详情** → Holder + 审批通过的申请者（Viewer）
3. **编辑/上传** → Holder + 核心成员（Member）
4. **管理权限** → 仅 Holder

### 2.4 时间戳防抄袭系统

**核心机制**: 当用户发布创意时，系统生成不可篡改的时间戳证明。

```
用户发布 Idea
    │
    ▼
计算 Hash = SHA256(标题 + 概述 + 创建者ID + 创建时间)
    │
    ▼
存储到数据库 (创建时间由服务器记录)
    │
    ▼
可选: 写入区块链/第三方时间戳服务 (扩展)
    │
    ▼
生成"创意证书"页面 (含Hash、时间、创建者)
```

**防抄袭举证流程**:
1. A 发现 B 的创意与自己的相似
2. A 提供自己的创意证书（含时间戳 Hash）
3. 平台可验证：A 的 Hash 对应的创建时间早于 B
4. 可作为学校/老师的仲裁依据

## 3. 系统架构

### 3.1 技术栈

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **前端** | React 19 + TypeScript + Vite + Tailwind CSS + shadcn/ui | 现代、类型安全、开发体验好 |
| **后端** | Node.js + Express + TypeScript | 生态丰富，JSON处理友好 |
| **数据库** | PostgreSQL (主) + Redis (缓存/会话) | 关系型数据为主，Redis加速 |
| **ORM** | Prisma | 类型安全，迁移方便 |
| **文件存储** | 本地 MinIO / AWS S3 兼容 | 支持文档预览、版本管理 |
| **实时通信** | Socket.io | 通知、在线协作状态 |
| **认证** | JWT (Access + Refresh Token) | 无状态、易扩展 |
| **容器化** | Docker + Docker Compose | 一键部署 |

## 6. 文件存储与文档版本控制

### 6.1 设计理念

**不做实时协同编辑，做版本快照管理。**

理由：
- 实时协同编辑需要 OT/CRDT 算法、WebSocket 状态同步、光标同步，实现复杂度高
- 大学生团队通常已有熟悉的在线编辑工具（飞书文档、腾讯文档、Google Docs）
- 平台的核心价值是"组队匹配"而非"文档编辑"
- 版本快照已能满足 90% 的协作需求：归档、追溯、回滚

### 6.2 支持的文件类型

| 类型 | 扩展名 | 预览方式 |
|------|--------|---------|
| Word | .doc, .docx | 转 PDF / Microsoft Office Viewer |
| PowerPoint | .ppt, .pptx | 转 PDF / 缩略图 |
| PDF | .pdf | 原生浏览器预览 |
| 图片 | .jpg, .png, .gif | 原生预览 |
| Markdown | .md | 渲染为 HTML |
| 代码文件 | .py, .js, .java 等 | 代码高亮 |

### 6.3 版本控制流程

```
团队成员本地/在线编辑文档
        │
        ▼
    上传文件到 TeamUp
        │
        ▼
┌─────────────────┐
│  文件类型校验    │  → 黑名单/白名单过滤
│  病毒扫描 (扩展) │
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  生成唯一存储Key │  → ideas/{ideaId}/documents/{docId}/v{version}.ext
│  计算文件 Hash   │  → 用于去重和完整性校验
└─────────────────┘
        │
        ▼
┌─────────────────────────────────────────┐
│           版本归档逻辑                   │
│  • 如果 Document 已存在当前版本          │
│  • 将当前版本复制到 DocumentVersion 表   │
│  • 更新 Document 为新上传的文件          │
│  • versionNumber + 1                     │
└─────────────────────────────────────────┘
        │
        ▼
   存储到 MinIO/S3
   bucket/ideas/{ideaId}/documents/{docId}/v2.docx
        │
        ▼
   数据库记录文档元数据
        │
        ▼
   生成预览 (异步)
   • Office 文档 → libreoffice 转 PDF
   • PDF → 直接提供
   • 图片 → 生成缩略图
```

### 6.4 版本回滚

```
用户选择回滚到 vN
        │
        ▼
┌─────────────────┐
│  从 DocumentVersion │
│  读取 vN 的存储Key  │
└─────────────────┘
        │
        ▼
┌─────────────────────────────────────────┐
│  将当前 Document 归档到 DocumentVersion │
│  用 vN 的数据覆盖 Document（成为最新版） │
│  versionNumber = 当前最大版本 + 1        │
│  comment = "回滚到 vN"                  │
└─────────────────────────────────────────┘
```

## 7. 安全设计

### 7.1 认证与授权

- **JWT**: Access Token (15分钟) + Refresh Token (7天)
- **密码**: bcrypt 哈希，盐值 12
- **RBAC**: 基于角色的权限控制
- **CORS**: 白名单限制

### 7.2 时间戳防抄袭

- 服务器端生成 Hash，不依赖客户端时间
- Hash 包含：标题 + 概述 + 创建者ID + 服务端时间戳
- 数据库记录创建时间，不可修改
- 提供公开验证页面，任何人可验证时间戳

### 7.3 数据安全

- **文档访问**: 只有创意核心成员可访问详细文档
- **只读防泄漏**: Viewer 级别用户只能在线预览，禁止下载；前端禁用右键/选择/复制
- **接口鉴权**: 每个 API 都校验当前用户权限
- **SQL注入**: Prisma ORM 自动防护
- **XSS**: 前端输出转义，CSP 策略

## 10. 关键设计决策记录 (ADR)

### ADR-001: 双轨空间设计

**决策**: 采用开放空间 + 课程空间的双轨制，而非单一模式。

**理由**:
- 大学生场景天然分为自由探索和课程约束两种
- 课程空间需要Owner角色（老师），开放空间去中心化
- 数据隔离清晰，避免课程数据混入公开广场

### ADR-002: 创意权限分层

**决策**: 公开概述 + 受控详情的分层权限。

**理由**:
- 降低发布门槛，鼓励分享创意
- 保护核心实现细节不被随意查看
- "申请-审批"机制建立初步信任

### ADR-003: 服务端时间戳 (非区块链)

**决策**: MVP 阶段使用服务端哈希 + 时间戳，不上链。

**理由**:
- 满足大学生场景的基本举证需求
- 成本极低，实现简单
- 未来可平滑迁移到区块链存证，作为增强层

### ADR-004: 版本快照替代实时协同编辑

**决策**: 用户在外部编辑工具中协作，TeamUp 负责版本归档与快照管理，不做实时协同编辑。

**理由**:
- 实时协同编辑（OT/CRDT）实现复杂度极高，偏离平台核心价值
- 大学生团队已习惯飞书/腾讯文档/Google Docs 等成熟工具
- 版本快照已满足绝大多数需求：归档、追溯、回滚、对比
- 将开发资源集中在组队匹配、权限管理、时间戳等核心功能上

**实现方式**:
- Document 表存储当前最新版本
- DocumentVersion 表存储历史版本快照
- 上传新版本时自动归档旧版
- 支持回滚到任意历史版本
